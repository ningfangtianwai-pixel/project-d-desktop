## 一、结论先行

### 1.1 Project D 不应被定义成“功能更多的桌面工具箱”

当前 V2 说明书提出了文件夹门户、AI 建议、全局搜索、待办、快捷键、布局快照等正确方向，但整体仍然是“Fences + 搜索 + 壁纸 + 桌宠 + AI”的功能相加。商业级产品不能只依靠模块数量形成优势，因为每个单项都已经存在成熟标杆：

- Fences 的优势是原生桌面整理、规则、门户、Peek 和布局恢复。[Fences 6 官方产品页](https://www.stardock.com/products/fences/index)
- Everything 的优势是确定、快速、低资源的本地文件检索。[Everything 官网](https://www.voidtools.com/)
- PowerToys Command Palette、Flow Launcher、Raycast 的共同特点是把大量操作收敛到统一快捷入口。[Command Palette 官方文档](https://learn.microsoft.com/en-us/windows/powertoys/command-palette/overview) [Flow Launcher 官网](https://www.flowlauncher.com/) [Raycast Windows 官方页](https://www.raycast.com/windows)
- Wallpaper Engine、Lively、Rainmeter 都把桌面个性化延伸到可创作内容或扩展体系。[Wallpaper Engine 官方 Steam 页面](https://store.steampowered.com/app/431960/Wallpaper_Engine/) [Lively 官方 GitHub](https://github.com/rocksdanister/lively) [Rainmeter 官方 GitHub](https://github.com/rainmeter/rainmeter)
- Desktop Mate 通过角色专属动作、声音和持续可见的窗口交互提供陪伴体验。[Desktop Mate 官方 Steam 页面](https://store.steampowered.com/app/3301060/Desktop_Mate/)
- Copilot、Raycast AI、screenpipe 分别展示了显式屏幕授权、AI 记忆和本地计算历史等上下文路径。[Copilot Vision 官方说明](https://support.microsoft.com/en-us/microsoft-copilot/using-copilot-vision-with-microsoft-copilot) [Raycast 隐私政策](https://www.raycast.com/privacy) [screenpipe 官方 GitHub](https://github.com/screenpipe/screenpipe)

Project D 若分别追赶这些产品，会变成复杂、耗资源、难以维护的集合。真正可形成差异化的方向应是：

> **Project D 是一个本地优先、可撤销、具有人格化界面的 AI 桌面管家。Luna 不是额外摆件，而是“理解桌面状态、提出计划、获得授权、执行动作、允许撤销”的统一交互层。**

### 1.2 建议采用“一条闭环、三个表面、四层壁垒”

**唯一核心闭环：**

```text
桌面状态变化
  → 本地规则识别机会
  → Luna 给出原因明确的建议
  → 用户预览将发生什么
  → 用户确认或修改
  → 执行并生成操作记录
  → 一键撤销
  → 本地学习偏好
```

**三个产品表面：**

1. **桌面层**：容器、门户、壁纸、天气、待办和自然桌面交互。
2. **命令层**：全局快捷入口，用于搜索、打开、整理、切换场景和调用动作。
3. **陪伴层**：Luna 的角色、状态、主动建议、对话和结果反馈。

**四层商业壁垒：**

1. **可信执行**：预览、确认、撤销、审计和恢复，比“会聊天”更重要。
2. **桌面语境**：理解文件、时间、天气、场景与用户习惯，但默认只处理必要元数据。
3. **人格化反馈**：同一行动通过动作、语气和环境变化被感知，形成品牌记忆。
4. **内容与扩展生态**：角色包、场景包、动作包和经过权限审核的扩展，而不是一开始开放任意代码插件。

### 1.3 对现有 V2 工程说明书的总判断

| 判断 | 结论 |
|---|---|
| 产品方向 | “AI 桌面伴侣”正确，应保留 |
| 第一突破口 | 应从“可信整理 + 即时搜索”切入，而不是同时铺开待办、看板和所有 AI 能力 |
| 差异化 | 不能只写“Fences + 搜索 + AI 陪伴”，必须落实为可撤销行动闭环 |
| 搜索方案 | 不建议从零实现全盘内存索引；优先适配 Everything，回退 Windows Search/受控目录索引 |
| AI 方案 | 本地规则先判断，LLM 负责理解和表达；不应把全部桌面文件直接发给云模型 |
| 商业化 | 不建议按容器数、待办数等方式削弱基础体验；Pro 应售卖高级自动化、场景、角色和跨设备价值 |
| 生态 | 应先定义签名包、权限声明和兼容策略，再考虑开放插件商店 |
| 工期 | 原文约 4 周可完成“功能原型”，不足以完成商业级可靠性、隐私、升级、安装和内容生产；建议 12-16 周进入可公开付费测试 |

## 二、研究方法与判断框架

本研究按六个维度观察产品，而不是只列功能：

1. **核心工作流**：用户为什么每天打开它，最短路径是什么。
2. **差异化**：它解决了什么其他产品没有解决好的问题。
3. **商业模式**：买断、订阅、DLC、免费开源、赞助或企业授权。
4. **隐私与本地化**：数据默认在哪里、云端何时介入、用户是否能关闭。
5. **性能与可靠性**：后台常驻、暂停策略、多显示器、睡眠恢复、错误隔离。
6. **生态与留存**：插件、内容、习惯、历史、角色资产或团队部署如何带来长期使用。

“值得借鉴”代表适合 Project D 的定位与现有技术基础；“不应照搬”代表会破坏可信度、性能、聚焦度或商业可持续性。

## 四、逐产品研究与 Project D 判断

### 4.1 Stardock Fences 6：桌面整理的基准不是“画几个框”

**官方事实**

- Fences 以桌面上的半透明分组承载原生应用、文件与文件夹，支持按文件类型、名称、时间和目标位置自动归类；Folder Portal 将 Explorer 中的目录内容带到桌面；Peek 用 `Win+Space` 将分组临时显示在其他窗口上方。[Fences 6 官方产品页](https://www.stardock.com/products/fences/index)
- Fences 还提供 Quick Hide、Roll-up、桌面页、布局快照、规则与容器个性化。官方历史材料明确说明快照可用于恢复自动布局，Folder Portal 可直接访问文件夹内容。[Fences 官方快速入门](https://www.stardock.com/blog/483309/getting-started-with-fences---3-part-quick-tips-video-series)
- Fences 长期更新重点大量集中于多显示器、DPI、休眠唤醒、网络驱动器、Explorer 兼容、门户滚动和布局恢复。这说明商业级桌面产品的主要成本在边缘状态可靠性，而不只在 UI。[Fences 官方更新历史](https://www.stardock.com/products/fences/history)
- 家庭版官方页面同时提供年度订阅和永久许可，并提供最多 5 台设备的多设备方案；企业部署支持 Intune、静默激活、预激活和集中许可管理。[Fences 6 官方产品页](https://www.stardock.com/products/fences/index) [Stardock Intune 部署说明](https://www.stardock.com/news/537913/deploying-stardock-products-with-intune)
- Stardock 产品需要联网激活，官方说明会记录姓名/邮箱（视购买方式而定）、序列号、IP、非个人机器信息和激活尝试次数。[Stardock 激活信息](https://www.stardock.com/support/activation-info)

**Project D 值得借鉴**

1. 将“原桌面的自然延伸”设为不可妥协原则，整理结果必须直接发生在用户认知中的桌面空间。
2. 门户、规则、Peek、快照不是四个孤立功能，而是“日常使用、临时访问、异常恢复”的完整链路。
3. 对多显示器、DPI、Explorer 重启、睡眠恢复、网络目录离线建立正式兼容矩阵。
4. 所有布局变化进入快照和撤销日志，不能只存当前结果。

**不应照搬**

1. 不应在 V2 直接深度修改 Explorer 原生图标坐标。Project D 当前可恢复的透明覆盖层路线更适合先建立安全与稳定。
2. 不应把“无限容器”作为主要付费点。整理是建立日常习惯的基础，过早限制会阻断留存。
3. 不应只模仿视觉上的框；如果没有规则命中解释、冲突处理和快照恢复，只会得到一个较漂亮的文件管理器。

### 4.2 Wallpaper Engine：真正的壁纸优势来自内容飞轮和性能合同

**官方事实**

- Wallpaper Engine 支持 2D/3D、视频、网页和应用型壁纸，支持交互、多显示器、屏保、编辑器和 Steam Workshop。其官方 Steam 页面当前列价 4.99 美元，并允许用户免费分享和下载 Workshop 内容。[Wallpaper Engine 官方 Steam 页面](https://store.steampowered.com/app/431960/Wallpaper_Engine/)
- 官方把性能控制作为核心卖点：其他应用全屏或游戏运行时可暂停或完全停止壁纸；应用规则还能在特定进程运行时释放壁纸内存。[Wallpaper Engine 应用规则与性能说明](https://help.wallpaperengine.io/en/performance/game.html)
- 官方创作文档要求网页壁纸遵循用户 FPS 上限，建议使用 `requestAnimationFrame`；纹理文档建议控制纹理显存，并以约 300 MB VRAM 或更低作为良好表现的参考。[FPS 限制文档](https://docs.wallpaperengine.io/en/web/performance/fps.html) [纹理优化文档](https://docs.wallpaperengine.io/en/scene/performance/texture.html)
- 官方 Steam 页面明确提醒：集成显卡可运行但专用 GPU 更推荐，网页和应用型壁纸通常比 2D/3D/视频占用更多资源。[Wallpaper Engine 官方 Steam 页面](https://store.steampowered.com/app/431960/Wallpaper_Engine/)

**Project D 值得借鉴**

1. 壁纸应是可导入、可打包、可预览、可降级的内容系统，而不是内置若干图片。
2. 动态壁纸、天气粒子和桌宠共享统一的“桌面渲染预算管理器”。
3. 支持前台遮挡、全屏、游戏、远程桌面、电池和低电量状态的自动暂停/降帧。
4. 场景包应包含壁纸、天气色调、容器主题、桌宠动作与环境音配置，形成 Project D 自己的内容单位。

**不应照搬**

1. 不应在 V2 复制完整壁纸编辑器，这会吞噬工程资源且与 Wallpaper Engine 正面竞争。
2. 不应默认支持任意网页作为壁纸；网页内容涉及网络、脚本、性能和安全边界。
3. 不应把高帧率粒子作为“高级感”的代名词。轻薄本优先采用高质量低帧率、事件驱动和预渲染素材。

### 4.3 Lively Wallpaper：免费开源产品仍然可以有商业级性能纪律

**官方事实**

- Lively 支持视频/GIF、网页、应用/游戏型壁纸、屏保、多显示器、命令行控制和开发 API；官方仓库称其完全免费开源且没有付费墙。[Lively 官方 GitHub](https://github.com/rocksdanister/lively)
- Lively 在全屏应用/游戏、使用电池和远程桌面时可以暂停壁纸，并支持按前台应用配置播放规则。[Lively 官方 GitHub](https://github.com/rocksdanister/lively)
- 近期官方版本采用按屏幕网格计算窗口遮挡的暂停算法，解决并排窗口、后台窗口遮挡和各显示器独立暂停等问题；同时将 WebView2 设为默认网页引擎以减少安装体积。[Lively 官方 Releases](https://github.com/rocksdanister/lively/releases)
- 官方深度壁纸功能说明模型推理可 100% 本地运行，不需要网络，也不向云端发送数据。[Lively 官方 Releases](https://github.com/rocksdanister/lively/releases)

**Project D 值得借鉴**

1. 借鉴其遮挡检测和按显示器暂停思路，而不是仅判断“当前是否全屏”。
2. 保持命令行/IPC 自动化能力，为未来快捷命令和扩展提供稳定入口。
3. 壁纸导入后复制到受控资源库，避免原路径被删除后内容失效。
4. 对每个场景记录格式、分辨率、FPS、预计资源级别和兼容能力。

**不应照搬**

1. Lively 的“几乎所有格式都可成为壁纸”适合工具型开源产品；Project D 商业版应优先提供经过验证的内容包。
2. 不应把开源 API 数量当生态，Project D 还需要签名、权限、版本兼容和故障隔离。

### 4.4 Rainmeter：扩展性强，但学习成本不应转嫁给普通用户

**官方事实**

- Rainmeter 是 Windows 桌面定制工具，采用 GPL-2.0；官方仓库还明确承诺，除非用户或安装/操作人员明确要求，程序不会向其他联网系统传输信息。[Rainmeter 官方 GitHub](https://github.com/rainmeter/rainmeter)
- Rainmeter 的长期扩展能力来自 Skin、Measure、Meter、Lua 和 C++/C# 插件。官方版本历史持续维护插件 API、多语言、Game Mode、唤醒动作和多显示器相关问题。[Rainmeter 官方版本历史](https://github.com/rainmeter/rainmeter-docs/blob/master/source/history.html)
- 官方提供单独的 Plugin SDK，表明插件 ABI、示例与开发者文档是生态基础设施，而不只是一个“加载脚本”按钮。[Rainmeter 官方 Plugin SDK](https://github.com/rainmeter/rainmeter-plugin-sdk)

**Project D 值得借鉴**

## 五、跨产品规律：优秀桌面产品为什么会长期留在电脑里

### 5.1 留存不是靠主动打扰，而是靠五种复利

| 留存机制 | 竞品表现 | Project D 应用 |
|---|---|---|
| 肌肉记忆 | Flow、PowerToys、Raycast 通过全局快捷键形成固定入口 | 一个全局入口，搜索/命令/AI 共享，结果 100 ms 级出现 |
| 空间记忆 | Fences 用固定布局、门户和快照形成“东西应该在哪里” | 容器、项目区、门户和场景稳定，不随机重排 |
| 速度信任 | Everything 让用户相信“输名字就能找到” | 搜索基础层必须确定，AI 只做语义转换与结果解释 |
| 情感依恋 | Desktop Mate 通过动作、声音和角色内容形成陪伴 | Luna 用动作反馈真实任务结果，降低气泡噪声 |
| 内容/扩展复利 | Wallpaper Engine、Rainmeter、Flow 通过内容与插件扩大边界 | 先做受控场景包/角色包，再逐步开放签名扩展 |

### 5.2 商业产品的最低门槛是“可信”，不是“聪明”

Project D 对文件拥有写操作能力，因此必须比聊天机器人更严格：

1. 整理前显示文件数量、目标位置、冲突与预计结果。
2. 执行中允许取消，并以事务日志记录每一步。
3. 执行后可一键撤销，重启后仍能恢复。
4. 不可逆操作单独确认，默认不提供“AI 自动删除”。
5. AI 无法决定时回退规则或要求选择，不能猜测。

### 5.3 “本地优先”必须变成产品界面，不是宣传词

建议在设置中提供 **数据与权限中心**：

| 数据域 | 默认策略 | 用户可见控制 |
|---|---|---|
| 文件名、扩展名、大小、时间 | 本地处理 | 可选择允许目录，支持排除路径 |
| 文件内容 | 默认不读取 | 单次授权、目录授权、随时撤销 |
| 对话历史 | 本地保存 | 查看、删除、禁用记忆 |
| AI 云请求 | 默认只发送当前输入 | 请求前显示将发送的上下文类型 |
| 屏幕画面 | 默认关闭 | 仅单次截图，不做持续采集 |
| 麦克风 | 默认关闭 | 仅按住说话或显式会话 |
| IP 定位 | 仅用于天气且可关闭 | 显示来源、城市并允许手动覆盖 |
| 使用统计/崩溃报告 | 默认征求同意 | 分开开关，可预览上传内容 |

## 六、对现有 `V2_工程说明书.md` 的具体修正

### 6.1 应保留的方向

| 原方向 | 结论 | 修正后的产品意义 |
|---|---|---|
| 文件夹门户 | 保留，P0/P1 | 高价值日常入口，不只是镜像列表 |
| AI 整理建议 | 保留，重构 | 本地规则发现机会，Luna 解释并请求授权 |
| 全局快捷键 | 保留，提前 | 作为统一命令层，不只为几个页面分配快捷键 |
| AI 桌面搜索 | 保留，重构 | Everything/Windows Search 提供索引，AI 解析自然语言 |
| 桌宠主动建议 | 保留，重构 | 事件驱动、节流、可解释、可关闭 |
| 容器配色 | 保留，降优先级 | 属于场景系统，不是竞争壁垒 |
| 布局快照 | 保留，提前 | 同时承担安全恢复与场景切换 |
| 新手引导 | 保留 | 必须用真实文件预览和撤销完成“信任激活” |
| 自由版/Pro | 保留商业分层 | 不以限制基础整理数量为主要手段 |

### 6.2 必须调整的工程方向

#### A. 不从零手写全盘索引

原方案用内存 `Map` 遍历 `desktop_files`，只覆盖已进入数据库的桌面记录，无法成为真正的全盘搜索，也没有索引更新、磁盘离线、权限、百万文件规模和恢复策略。

**替代建议：**

```text
SearchProvider 接口
  ├─ EverythingProvider（优先，用户已安装或同意安装）
  ├─ WindowsSearchProvider（系统回退）
  └─ ScopedLocalProvider（仅桌面/下载/授权目录）

NaturalLanguageParser
  └─ 将“上周三改过的报价 PDF”解析为结构化条件

SearchRanker
  └─ 文本匹配 + 最近使用 + 当前项目 + 固定项
```

#### B. 不在 Renderer 用 `setInterval` 驱动主动建议

Renderer 可能重载、隐藏或冻结。主动建议应由主进程的事件总线与策略引擎触发，并持久化冷却时间、被拒绝次数和“今日不再提醒”。

#### C. AI 不直接分析整份文件列表

先由本地规则生成结构化事实，例如：

```json
{
  "unclassifiedCount": 12,
  "newToday": 7,
  "duplicateNameGroups": 2,
  "largeFiles": 1
}
```

云端模型只在用户允许时接收必要摘要，用于意图解析或人格化表达；执行计划由确定性代码验证。

#### D. 门户不能只依赖 `fs.watch + 30 秒轮询`

需要处理事件合并、重命名、删除、离线、无权限、网络路径、睡眠恢复、Watcher 溢出和重新全量同步。门户还必须明确“这是镜像，不是复制”，避免用户误解。

#### E. 数据看板不能直接聚合日志文本

日志用于诊断，不是业务事实源。整理、搜索、建议、撤销、场景切换应写入结构化本地事件表，再由看板查询。统计默认本地，不因“做增长”而强制上传。

#### F. 快捷键需要冲突治理

快捷键设置必须显示注册状态、冲突原因、替代建议和一键恢复默认。不能假设 `Win+Shift+...` 在所有系统、键盘布局和软件中可用。

### 6.3 原方案缺失、但商业化必须补齐的模块

1. 操作预览与冲突解决。
2. 跨重启撤销日志和恢复中心。
3. 数据与权限中心。
4. 搜索提供商适配层。
5. 统一动作模型与风险分级。
6. 桌面事件总线与建议节流策略。
7. 渲染预算管理器。
8. 自动更新、签名、迁移与回滚。
9. 崩溃恢复、诊断包和用户可读错误页。
10. 多显示器/DPI/Explorer/睡眠/远程桌面兼容矩阵。
11. 场景包、角色包和扩展包规范。
12. 素材版权、来源、版本和商业授权登记。

## 七、Project D V2 新版产品方案

### 7.1 产品定位

**一句话定位：**

> Project D 是 Windows 上本地优先的 AI 桌面管家：它让文件更容易找到、桌面更容易恢复、环境更有生命力，并且每一次改动都先预览、可撤销。

**首要目标用户：**

1. 桌面和下载目录经常堆积文件的知识工作者、学生与内容创作者。
2. 喜欢桌面美化和角色陪伴，但不愿牺牲性能的轻薄本用户。
3. 希望 AI 帮忙行动，但担心隐私和误操作的普通 Windows 用户。

**不作为 V2 首发目标：**

1. 企业大规模终端管理。
2. 持续屏幕/音频记录型个人知识库。
3. 通用自动化平台。
4. 专业动态壁纸编辑器。
5. 任意代码插件宿主。

### 7.2 三个主工作流

#### 工作流 A：安全整理

```text
Luna：“桌面新增了 11 个文件，其中 7 张截图。要看看整理方案吗？”
  → 打开预览
  → 展示每个文件将进入哪个容器/目录
  → 标出重名、占用和无权限项
  → 用户修改或确认
  → 执行
  → 显示“已整理 10 个，1 个跳过”
  → 30 秒内突出显示“撤销”，之后长期保留在恢复中心
```

#### 工作流 B：即时查找与行动

```text
Ctrl+Shift+Space
  → 输入“上周的报价 PDF”
  → 100 ms 级先显示本地候选
  → 语义解析补充日期/类型
  → Enter 打开
  → 右侧动作：定位、固定、加入项目区、询问内容
```

#### 工作流 C：桌面场景与陪伴

```text
选择“雨夜专注”场景
  → 壁纸、粒子、容器主题、Luna 服装/动作、提醒频率一起变化
  → 检测全屏/电池后自动降帧或静止
  → Luna 只在任务完成、天气变化或明确提醒时出现
```

### 7.3 统一动作模型

所有桌面操作都归一为动作对象：

```typescript
interface DesktopAction {
  id: string;
  type: "open" | "move" | "group" | "rename" | "apply-scene" | "restore";
  title: string;
  reason: string;
  risk: "read" | "reversible-write" | "destructive" | "external";
  inputs: ActionInput[];
  preview: ActionPreview;
  permissions: PermissionScope[];
  reversible: boolean;
}
```

风险规则：

| 风险级别 | 示例 | 行为 |
|---|---|---|
| 只读 | 搜索、预览、打开 | 可直接执行 |
| 可逆写入 | 移动、归组、改壁纸 | 首次或批量时确认，必须写撤销日志 |
| 破坏性 | 删除、覆盖、清空 | V2 默认不允许 AI 自动执行，必须二次确认 |
| 外部 | 发起网络请求、调用第三方程序 | 显示目标服务和将传输的数据 |

### 7.4 Luna Brain：规则、意图和人格三层分离

```text
Observation Layer（本地）
  文件事件、桌面状态、时间、天气、用户操作历史

Policy Layer（确定性）
  触发条件、冷却、权限、风险、可用动作、撤销能力

Intent Layer（本地解析优先，LLM 可选）
  把自然语言转为结构化搜索或动作

Personality Layer
  将事实和结果转为符合人格的短句与动作
```

关键原则：人格只改变表达，不改变权限和事实；LLM 可以提出候选计划，但不能绕过动作验证器。

### 7.5 场景系统代替零散美化设置

一个 Scene Pack 建议包含：

- 壁纸资源和预览图。
- 天气/粒子预设与性能档位。
- 容器颜色、透明度和字体对比度。
- Luna 服装、动作映射和可选声音。
- 可选待办样式和环境音。
- 兼容版本、版权、作者、资源预算和签名。

用户可以单独覆盖其中一项，但默认一键得到完整视觉结果。这样既能提高审美一致性，也为内容商业化建立清晰单位。

## 八、商业模式建议

### 8.1 不建议照搬原文的数量限制

原方案把容器、门户、待办、快照、颜色、人格和壁纸都按数量切割。这样容易让用户在尚未形成习惯前就感到“处处受限”，也会把产品价值误导为功能解锁。

建议按价值层分级：

| 层级 | 建议内容 | 商业目的 |
|---|---|---|
| Free | 安全整理、基础搜索、基础门户、1 套自动快照、基础 Luna、基础场景、完整撤销 | 让用户建立信任和日常习惯 |
| Pro | 高级自动化规则、自然语言动作、多个布局/项目场景、高级搜索、历史洞察、多提供商 AI、同步/导出能力 | 为高频效率和个性化收费 |
| Content | 角色包、动作/声音包、场景包、联名 IP 包 | 低边际成本内容收入与情感留存 |
| AI Credits | 可选托管模型额度；BYOK 用户不重复购买模型成本 | 隔离不稳定的推理成本 |
| Team（后续） | 部署模板、策略、扩展白名单、集中许可、支持 SLA | V2 验证后再进入企业市场 |

### 8.2 定价不是现在拍脑袋决定，应做三个实验

以下是**研究建议的测试区间，不是最终售价**：

1. **Pro 年费测试**：人民币 68/98/128 元三档，观察试用转化和续费意愿。
2. **早期永久版测试**：人民币 168-228 元，仅限明确的版本更新范围，不承诺无限期云 AI。
3. **内容包测试**：普通原创角色/场景 18-48 元；高成本语音或正版 IP 独立定价。

定价依据不是功能数量，而是三个可测价值：每周节省的查找时间、减少桌面混乱的频率、用户对角色/场景的付费偏好。

### 8.3 推荐收入结构

```text
基础产品：免费建立规模与信任
效率升级：Pro 订阅或永久许可
情感内容：角色/场景 DLC
推理成本：BYOK 或独立 Credits
企业能力：成熟后按设备/年授权
```

[Wallpaper Engine](https://store.steampowered.com/app/431960/Wallpaper_Engine/) 采用一次性购买并通过 Workshop 扩大内容，[Desktop Mate](https://store.steampowered.com/app/3301060/Desktop_Mate/) 采用免费宿主和角色 DLC，[Raycast](https://www.raycast.com/pricing) 则提供免费核心与 AI/同步相关 Pro 计划。基于这些官方商业模式，本文分析认为 Project D 更适合混合模式，而不是把所有模块塞进单一订阅。

## 九、隐私、安全与可信执行规范

### 9.1 必须公开的隐私承诺

1. 默认不上传文件名、路径和文件内容。
2. 默认不持续录屏、不持续录音。
3. 云 AI 请求显示提供商、将发送的数据类型和上下文范围。
4. API Key 使用系统安全存储或加密存储，不写日志、不随诊断包导出。
5. 用户可删除对话、偏好、索引、操作历史和全部本地数据。
6. 关闭 AI 后，整理、搜索、壁纸、天气、桌宠和恢复仍可使用。
7. 第三方扩展单独授权，宿主权限不自动继承给插件。

### 9.2 AI 记忆分层

| 记忆类型 | 默认 | 生命周期 | 示例 |
|---|---|---|---|
| 会话上下文 | 开启 | 最近 10 条或受控字符预算 | 当前对话连续性 |
| 人格偏好 | 开启 | 持久，可编辑 | 用户喜欢简洁提醒 |
| 整理偏好 | 经确认后学习 | 持久，可回滚 | 截图通常进“素材” |
| 文件内容记忆 | 关闭 | 单次或明确授权 | 某文档摘要 |
| 屏幕上下文 | 关闭 | 单次会话，结束即清除 | 对当前错误截图提问 |

### 9.3 恢复中心

恢复中心应成为商业版的核心页面，至少提供：

- 最近操作及每步状态。
- 一键撤销/重做。
- 自动布局快照和手动场景快照。
- 冲突、失败和跳过项。
- Explorer/壁纸宿主/桌宠窗口恢复状态。
- 导出诊断包前的内容预览与隐私脱敏。

## 十、性能与可靠性基线

以下是 Project D 的**建议验收目标**，需要在指定参考轻薄本上实测，不是竞品官方指标：

| 指标 | V2 商业测试门槛 |
|---|---|
| 冷启动到可交互 | P95 ≤ 3 秒 |
| 热启动/托盘恢复 | P95 ≤ 1 秒 |
| 本地搜索首批结果 | P95 ≤ 150 ms；Everything 可用时目标 ≤ 100 ms |
| 空闲 CPU | 10 分钟平均 ≤ 1%，无持续无意义轮询 |
| 空闲内存 | 默认三个桌面表面合计目标 ≤ 450 MB，逐进程有预算 |
| 动态场景 | 支持静止/15/30 FPS；轻薄本默认 30 FPS 或自适应 |
| 全屏/游戏/远程桌面 | 壁纸和粒子暂停或停止，渲染占用接近 0 |
| 电池模式 | 默认降低粒子、帧率和桌宠动画；低电量停止动态层 |
| 桌面整理 | 100% 生成操作记录；可逆操作重启后仍可撤销 |
| 崩溃恢复 | 主界面、桌宠、壁纸宿主分别故障隔离，不互相拖垮 |
| Crash-free sessions | 公开付费测试目标 ≥ 99.5% |
| 多显示器 | 1/2/3 屏，100%-250% DPI，插拔/休眠/主屏切换均有测试 |

### 10.1 渲染降级阶梯

```text
高质量：30 FPS + 完整粒子 + 完整角色动作
平衡：20 FPS + 减少粒子 + 降低桌宠刷新
省电：15 FPS + 静态天气层 + 简化动作
最低：静态壁纸 + 无粒子 + 桌宠低频待机
暂停：全屏/游戏/远程桌面/极低电量
```

系统应自动选择并允许用户锁定档位，切换时给出简短原因，不把性能问题留给用户猜测。

## 十一、扩展与内容生态路线

### 11.1 V2 不直接开放任意插件商店

商业首发先支持两种受控包：

1. **Scene Pack**：资源和声明式配置，不执行任意代码。
2. **Action Pack**：调用 Project D 已授权 API 的声明式动作，可配置输入、结果页和权限。

后续再增加受限脚本运行时。所有包必须包含：

```json
{
  "id": "author.package",
  "version": "1.0.0",
  "minProjectDVersion": "2.0.0",
  "type": "scene | action | character",
  "permissions": ["weather.read", "files.metadata.read"],
  "resourceBudget": { "fps": 30, "memoryMB": 120 },
  "signature": "...",
  "license": "commercial-or-open-source-id"
}
```

### 11.2 生态治理原则

1. 官方精选优先于无审核数量。
2. 权限在安装前显示，更新新增权限需再次确认。
3. 包故障自动隔离并可一键禁用。
4. 网络、文件写入、Shell 和 AI 调用分别授权。
5. 资源超预算可自动降级或暂停。
6. 每个资源必须记录作者、来源、许可证和商业使用范围。

## 十二、12-16 周商业级路线图

> 假设：单主开发者 + AI 协作 + 用户提供最终角色/壁纸素材。若有专职设计、测试或内容人员，可并行缩短；原说明书的 4 周更适合作为功能原型周期。

### Phase 0：商业底座（第 1-3 周）

**目标：先让产品值得信任。**

- 多显示器/DPI/休眠/Explorer 测试矩阵。

**阶段闸门：** 没有可跨重启撤销，就不进入 AI 自动整理。

**阶段闸门：** 搜索 P95 达标；门户离线、重命名、睡眠恢复可控。

### Phase 2：Luna 可信智能（第 8-10 周）

**目标：AI 能帮忙，但不会擅自做事。**

**阶段闸门：** 所有写动作通过验证器；云请求可观察、可关闭。

### Phase 3：视觉与陪伴商业化（第 11-13 周）

**目标：形成可售卖且不伤性能的品牌体验。**

- Luna 动作状态机、素材切片和天气/任务反馈。
- 角色/场景内容管理、预览、卸载和版权记录。
- 前台遮挡、电池、全屏、远程桌面自动降级。
- 新手引导围绕“预览整理 → 执行 → 撤销 → 搜索 → 选择场景”。

**阶段闸门：** 参考轻薄本性能达标；无素材裁剪、黑边或焦点抢占。

### Phase 4：付费测试准备（第 14-16 周）

**目标：可以向真实用户收费并承担支持责任。**

- 隐私政策、数据说明、素材许可证、第三方组件清单。
- 小规模公开测试和价格实验。

## 十四、需要持续追踪的产品指标

### 激活指标

### 信任指标

- 整理后立即撤销率。
- 文件冲突率、失败率和恢复成功率。
- 用户关闭 AI/主动建议的比例。
- 用户手动清除记忆或权限的频率。
- 与误操作、隐私、资源占用相关的支持请求率。

### 质量指标

- Crash-free sessions。
- 启动、搜索和整理耗时 P50/P95。
- WallpaperHost 附着/恢复成功率。
- 每个渲染档位的 CPU/GPU/内存分布。
- 多显示器和睡眠恢复失败率。

## 十五、建议立即确定的产品决策

1. 将产品正式定位从“AI 桌面伴侣功能集合”收敛为“本地优先、可撤销的 AI 桌面管家”。
2. V2 第一主线确定为“可信整理 + 即时搜索”，场景和桌宠负责品牌化反馈。
3. 放弃从零实现全盘文件索引，建立 SearchProvider 适配层。
4. 建立统一动作模型，AI 不能绕过动作验证和权限层。
5. 把恢复中心、权限中心、性能管理提前到功能开发之前。
6. Free 不阉割撤销、基础搜索和基础整理；Pro 收费点转向高级自动化与场景。
7. V2 只开放受控 Scene/Action/Character Pack，不开放任意代码插件。
8. 将 4 周目标改为“核心功能原型”，商业付费测试按 12-16 周规划。
9. 角色和壁纸素材建立授权台账，未授权资源只用于本地开发占位。
10. 任何新增功能都必须回答：它是否提高高频入口、可信执行、情感品牌或内容复利中的至少一项。

## 十七、最终建议

Project D 的最大商业机会，不是成为“比别人多几个模块”的桌面软件，而是占据一个目前仍然不够成熟的位置：

> **在不牺牲隐私、不破坏原生桌面、不要求高端硬件的前提下，让 AI 真正帮助普通用户整理、查找、恢复和塑造自己的桌面。**

[Fences](https://www.stardock.com/products/fences/index) 持续销售桌面整理能力，[Everything](https://www.voidtools.com/) 以即时本地检索为核心，[Wallpaper Engine](https://store.steampowered.com/app/431960/Wallpaper_Engine/) 与 [Desktop Mate](https://store.steampowered.com/app/3301060/Desktop_Mate/) 分别围绕壁纸和角色内容建立消费，[Raycast](https://www.raycast.com/windows) 与 [Command Palette](https://learn.microsoft.com/en-us/windows/powertoys/command-palette/overview) 则以统一入口承载扩展。基于这些事实，Project D 应把相关规律收束到 Luna 的可信行动闭环中，而不是逐项复制功能。

下一版工程规格应以“动作模型、撤销恢复、搜索适配层、权限中心、性能预算”作为骨架，再把门户、AI、场景和桌宠挂在这套骨架上。这样做看起来比继续堆功能慢，但它才是从漂亮原型走向可收费、可维护、可承担用户文件安全责任的路径。
