# Project D 支付接入与权益运行手册

> 状态：Gate 5 工程草案。本文不是支付渠道审核通过证明，也不是法律或财税意见。任何真实支付接入必须先取得合规商户主体、渠道审核、HTTPS 回调域名、服务端基础设施和律师/财务确认。

## 1. 当前边界

本仓库目前只提供 `server/` 下的无框架 TypeScript 领域骨架与内存适配器：

- Account、Device、Order、Payment callback、Entitlement、Audit 接口；
- 服务端产品目录定价；
- 订单状态机；
- 回调验签接口和事件幂等；
- 退款撤权；
- 短期签名权益快照抽象；
- 重复回调、客户端越权和退款撤权测试。

它**不是可直接暴露到互联网的支付服务器**。当前没有 HTTP 层、数据库事务、登录认证、限流、渠道 SDK、KMS、队列、对账任务、监控或灾备。

## 2. 不可违反的安全规则

1. 客户端只能请求创建订单和查询订单，不得提交或更新 `paid`、`refunded` 等状态。
2. 金额、币种、套餐和权益必须从服务端产品目录读取，不能相信客户端传值。
3. 支付成功只能来自通过渠道官方规则验签的服务端通知或服务端主动查单。
4. 每个渠道事件按 `channel + eventId` 去重，权益再按 `sourceOrderId` 防重复开通。
5. 回调验签、订单迁移、权益变更和审计写入必须放在一个数据库事务或等效的可靠消息流程中。
6. 支付私钥、商户密钥、管理员凭据和权益签名私钥只存放在 KMS/HSM 或受控服务端密钥系统。
7. 客户端只保存短期签名权益快照，正式版内置公钥，不内置可签发权益的对称密钥或私钥。
8. 日志禁止记录完整回调报文、Authorization、银行卡数据、支付密钥和用户密码。

## 3. 渠道选择

### 支付宝和微信

必须使用具备 API、异步通知、服务端验签、退款和对账能力的商户产品。个人收款码、个人转账、监听到账通知或要求用户填写备注不能形成可靠商业闭环，也难以满足退款、对账、税务和风控要求，不应接入自动发放权益。

### 银行卡

应通过持牌银行或支付服务商的托管收银台/Token 化方案接入。Project D 服务端不接收、不保存完整卡号、CVV、磁道信息或支付密码，尽量降低 PCI DSS 范围。是否允许个人主体接入由收单机构和当地法律决定。

## 4. 推荐部署边界

```text
Electron client
  -> HTTPS API gateway
    -> Account/Order service
      -> transactional database
      -> payment channel adapter
      -> entitlement service
      -> append-only audit store
      -> outbox/event worker

Payment provider
  -> dedicated HTTPS callback endpoint
    -> raw-body capture with size limit
    -> timestamp/replay checks
    -> official signature verifier
    -> idempotent domain transaction
```

生产环境应把 HTTP/API、领域层、持久化适配器和渠道适配器分开。`server/in-memory.ts` 仅用于本地测试，不能作为生产持久化。

## 5. API 最小契约

建议客户端 API：

- `POST /v1/orders`：请求体只包含 `sku`、`channel`、客户端幂等键；账号来自服务端会话。
- `GET /v1/orders/{orderId}`：只允许订单所属账号查询。
- `POST /v1/devices/register`：注册安装标识，执行设备数量和风险策略。
- `DELETE /v1/devices/{deviceId}`：解绑并吊销该设备会话。
- `GET /v1/entitlements/snapshot?deviceId=...`：返回短期签名快照。
- `POST /v1/account/logout-all`：吊销全部会话。

渠道回调 API 必须独立，例如 `POST /callbacks/alipay`、`POST /callbacks/wechat-pay`。回调接口不使用用户会话，但必须验证渠道签名、商户号、时间戳、nonce、金额、币种、订单号和事件类型。

服务端响应中不要根据客户端传入的金额或 `status` 字段改变订单。未知字段应忽略或拒绝，并记录受控安全指标。

## 6. 订单状态机

```text
pending -> paid | failed | cancelled | expired
failed/cancelled/expired -> paid       (仅允许验签后的延迟到账或主动补单)
paid -> refunded
refunded -> terminal
```

相同状态重复通知是幂等操作。`refunded -> paid`、`pending -> refunded` 等迁移必须拒绝并进入人工核对。部分退款、分期和多次扣款不在当前骨架范围内，正式支持前应扩展独立退款实体，不能复用全额退款状态冒充。

## 7. 创建订单

1. 验证账号会话和设备状态。
2. 使用客户端幂等键查询已有订单。
3. 从服务端产品目录读取 SKU、金额、币种和权益定义。
4. 在数据库事务内创建 `pending` 订单和审计记录。
5. 调用渠道下单接口，保存渠道订单号或预支付标识。
6. 把渠道要求的拉起参数返回客户端，但不返回服务端密钥。
7. 客户端支付完成后只轮询订单状态，不自行开通会员。

## 8. 回调处理

处理顺序不得改变：

1. 限制请求方法、来源速率、报文大小和读取超时。
2. 保留渠道要求的原始字节，禁止先 JSON 重排再验签。
3. 使用官方 SDK/文档验证签名、证书链、时间戳、nonce、商户号和应用号。
4. 将回调转换成内部 `VerifiedPaymentEvent`；未验签数据不得进入领域层。
5. 开启数据库事务并锁定订单。
6. 插入唯一键 `(channel, event_id)`；冲突则返回渠道要求的成功 ACK，不重复开通。
7. 核对订单渠道、金额、币种和渠道订单号。
8. 迁移订单状态，并按 `source_order_id` 幂等授予或撤销权益。
9. 写追加式审计和 outbox 事件。
10. 提交事务后再 ACK；业务失败返回渠道规定的可重试响应。

骨架中的内存仓储会在处理异常时释放事件占位以允许重试。生产数据库必须用唯一约束、行锁和事务实现，不能依赖单进程 Map。

## 9. 权益签名

生产建议使用 KMS 托管的 Ed25519 或 ECDSA 私钥：

- 快照包含账号、设备、套餐、能力、签发时间、过期时间、离线宽限和权益版本；
- 私钥只允许 Entitlement 服务调用；
- 客户端内置当前和下一把公钥，可按 `keyId` 轮换；
- 在线快照建议 5-15 分钟过期，离线宽限应由套餐和风险策略决定；
- 退款、拒付、管理员撤销和设备解绑会提升权益版本，使旧快照在重新联网后失效；
- 客户端本地数据库的 `tier=pro` 或功能开关不能代替有效签名。

`InMemoryHmacEntitlementSnapshotSigner` 只用于单元测试，绝不能把其共享 secret 放进正式客户端。

## 10. 退款、拒付与风险订单

- 全额退款：订单从 `paid` 进入 `refunded`，撤销该订单产生的全部有效权益。
- 重复退款通知：按事件号和订单号幂等，不重复撤销。
- 拒付/争议：创建独立风险事件，按合同与法律要求暂停或撤销权益，保留申诉入口。
- 退款失败：订单仍保持原状态，记录渠道失败原因并重试或人工处理。
- 已消费数字服务的退款规则必须在购买前展示，并由律师审查；工程状态不能替代消费者权利。

## 11. 主动查单与对账

异步回调不是唯一证据。应建立：

- 延迟订单查单：对长时间 `pending` 的订单调用渠道服务端查询；
- 日对账：下载渠道账单，与订单、退款和权益逐项核对；
- 差异队列：金额不符、渠道成功但本地未开通、本地开通但渠道无单均进入人工处理；
- 补偿任务：使用同一领域命令和幂等键修复，禁止直接改数据库状态；
- 指标：支付成功率、到账延迟、重复回调、验签失败、差异单、退款撤权延迟。

## 12. 密钥与日志

- 密钥分别配置开发、测试、生产环境，禁止共用。
- CI 只引用密钥标识，不输出密钥值；仓库运行 secret scan。
- 回调验签和权益签名密钥至少支持双钥轮换与紧急吊销。
- 管理员操作需要强认证、最小权限和不可覆盖审计。
- 日志只记录内部订单号、事件号散列或受控标识、状态和错误码。
- 数据库备份加密，恢复演练要验证订单、回调幂等键、权益和审计同时恢复。

## 13. 测试矩阵

- [x] 重复成功回调不重复开通（内存领域测试）。
- [x] 客户端伪造金额或 `paid` 状态无效（内存领域测试）。
- [x] 未验签回调被拒绝（内存领域测试）。
- [x] 验签退款撤销对应权益（内存领域测试）。
- [ ] 渠道官方沙箱：成功、失败、取消、超时和延迟到账。
- [ ] 同事件并发 100 次，仅提交一次事务。
- [ ] 不同事件号重复通知同一成功订单，不重复权益。
- [ ] 回调重放、过期时间戳、错误商户号、金额不符和证书轮换。
- [ ] 断网、数据库死锁、队列重复投递和 ACK 丢失后的恢复。
- [ ] 全额退款、退款失败、拒付、争议和人工补单。
- [ ] 设备超限、换机、到期、永久权益和离线宽限。
- [ ] 渗透测试、限流、灾备、KMS 轮换和审计导出。

## 14. 上线门禁

以下任一缺失时保持支付入口关闭：

- 合法商户主体和渠道书面审核；
- 正式域名、TLS、生产数据库、备份和监控；
- 官方验签适配器及渠道沙箱证据；
- KMS 非对称权益签名；
- 隐私政策、用户协议、会员/退款/自动续费文本律师审核；
- 账务对账、客服查询、退款和事故响应流程；
- 真实 P0/P1 演练与负责人签字。
