# Project D V3.0 商业化落地与质量门禁计划

版本：V3.0  
日期：2026-07-16  
适用平台：Windows 10 / Windows 11  
状态：执行基线

## 1. 执行结论

Project D 已经从产品原型进入“客户端功能较完整、商业证据不足”的阶段。V3 不再用页面数量或功能开关衡量完成度，而以四类结果衡量：

1. 桌面宿主在崩溃、休眠、显示器变化和异常退出后可以自动恢复。
2. 动态壁纸在轻薄本、多屏和高 DPI 环境中有可复现的性能预算。
3. 安装、升级、回滚、签名、账号、支付和权益由可信链路支撑。
4. 每一张商业资源均有授权证据，用户数据处理有正式政策和删除机制。

当前发布结论仍为 **NO-GO（正式付费版）**，可以继续作为内部测试版或封闭 Beta。V2.1 客户端功能口径约 91%，严格商业准备度在本轮 P0 修复后约 37%；该数字不会因为新增页面而上升，只会在质量门禁取得真实证据后上升。

## 2. V3 产品定位

### 2.1 核心承诺

Project D 是“桌面空间助手”，不是传统文件管理器，也不是单一壁纸播放器。

- 桌面整理直接发生在 Windows 桌面语境中，动作先预览、可撤销、可恢复。
- 壁纸、天气和桌宠是桌面的自然延伸，不遮挡系统图标和日常应用。
- Luna 可以理解上下文、建议行动和切换本地资源，但不能越过用户确认执行高风险动作。
- 产品优先保证稳定、低占用和隐私，再扩展内容社区与生成式能力。

### 2.2 V3 首发边界

- 仅发布 Windows x64 正式版；清理 macOS/Linux 打包声明。
- 首发支持本地图片、受控视频壁纸和本地壁纸库。
- 不在首发宣传 AI 生成壁纸、UGC 社区或跨平台同步。
- 账号与支付上线前，免费 Beta 不设置可被本地篡改的伪会员限制。
- 客户端不内置公司级 AI Key、支付密钥或管理员凭据。

## 3. 不可降低的发布原则

1. 任一 P0 未关闭，不发布付费正式版。
2. “代码已实现”不等于“验收通过”；必须有真实安装包、硬件或服务端证据。
3. 自动恢复不能以删除用户数据、重启 Explorer 或隐藏桌面图标为默认代价。
4. 支付结果和会员权益必须由服务端签名确认，客户端只保存短期缓存。
5. 未取得可审计授权的素材不得进入商业安装包。
6. 更新失败必须回到上一可运行版本，不能把恢复责任交给用户。
7. 低配机器达不到动态效果预算时，应自动降级粒子、帧率或视频，而不是卡住桌面。

## 4. 商业质量门禁

## Gate 0：桌面外壳可靠性

目标：消除白屏、无法关闭和渲染进程失联导致的桌面不可用风险。

当前已完成：

- Electron 运行基线已升级并锁定到 43.1.1。
- 主窗口、设置、覆盖层、壁纸和桌宠统一监听渲染崩溃、加载失败、预加载错误和无响应。
- 休眠唤醒、解锁、显示器插拔和 DPI 变化后执行 DOM 尺寸健康探针。
- 单次故障自动重载；短时间连续失败有次数上限，并升级为禁用硬件加速的安全重启。
- 壁纸重载后重新挂接桌面宿主；桌宠重载后恢复置顶、全工作区和鼠标穿透。
- 退出清理设置 8 秒硬截止；超时写日志并强制结束，不再留下不可关闭窗口。
- 已通过真实 Electron 渲染崩溃注入和退出永久挂起注入。

关闭条件：

- 最终安装包再完成一次崩溃恢复与正常关闭实测。
- 20 次锁屏/解锁和 20 次睡眠/唤醒无持续白屏。
- 任何异常后桌面图标、原系统壁纸和 Explorer 均处于可用状态。
- 错误日志能区分 renderer、GPU、preload、加载失败和超时退出。

## Gate 1：壁纸引擎 2.0

目标：把“支持图片的桌面宿主”升级为稳定的静态/动态壁纸引擎。

本 Gate 分两波执行，降低 Codex 单次上下文复杂度：

### W1A：媒体运行时

交付项：

- 建立统一媒体模型：图片、视频、时长、分辨率、编码、循环、授权和降级资源。
- 引入授权视频样本，最低基线：1080p H.264、2K H.264、4K H.264；H.265 和 VP9/WebM 作为可选能力测试，不作为首发核心依赖。
- 增加显式播放、暂停、继续、循环和静音状态，不以销毁窗口代替暂停。
- 建立 `PauseArbiter`，统一处理用户暂停、全屏应用、锁屏、休眠、电池和系统节能原因。
- 全屏、锁屏、休眠、电池事件接入视频、Pixi 和天气刷新。
- 更换媒体时保留上一帧，解码失败回退静态封面，禁止黑色闪屏。

### W1B：多屏与系统恢复

交付项：

- 每显示器保存独立壁纸、裁剪策略和性能档位；拔屏后保留映射，重接后恢复。
- 热插拔后壁纸状态正确恢复。
- 增加 Windows 登录自启动开关，区分“应用自启动”和“启动后自动接管桌面”。
- 安装器恢复脚本复用运行时 Explorer 图标恢复实现。

### 门禁指标（W1A + W1B 合并验收）

- 视频连续循环 100 次，无黑帧、音频泄漏和不可恢复解码失败。
- 连续切换 300 次，无窗口丢失、宿主脱离或内存持续增长。
- 软件正常退出、强杀、注销和重启后，系统桌面均可恢复。
- 全屏应用出现后 2 秒内暂停，退出全屏后按策略恢复。
- 多屏热插拔 20 次，壁纸映射正确恢复。

### 视频测试矩阵

| 样本 | 编码 | 用途 | 必测/可选 |
|---|---|---|---|
| 1080p H.264 | H.264 | 最低兼容基线 | 必测 |
| 2K H.264 | H.264 | 中档基线 | 必测 |
| 4K H.264 | H.264 | 高负载兼容基线 | 必测 |
| 4K H.265 | H.265 | 可选能力 | 可选 |
| 4K VP9/WebM | VP9 | 可选能力 | 可选 |

首发核心能力不依赖用户系统是否安装额外编解码组件。

## Gate 2：性能与自适应质量

目标：给“低占用”建立可测量标准，并让程序主动适应轻薄本。

交付项：

- 增加本地运行指标采样：主进程、renderer、GPU、视频解码、工作集、帧率和掉帧。
- 设置四档质量：静态、节能、均衡、高质量；自动档根据电池、全屏和负载切换。
- 页面隐藏或壁纸暂停时停止 Pixi ticker、视频解码、天气刷新和非必要计时器。
- 纹理、视频和缩略图使用有上限的 LRU 缓存，切换后主动释放 GPU 资源。
- 建立 4 小时交互长测和 24 小时空闲长测脚本，输出 CSV/JSON 报告。

参考机器门槛（1080p、Intel 核显轻薄本）：

| 场景 | 目标 |
|---|---|
| 静态壁纸稳定待机 | CPU 中位数不高于 1%，P95 不高于 3% |
| 1080p 动态壁纸 | CPU 中位数不高于 5%，无持续掉帧 |
| 全屏暂停 | 壁纸 renderer 接近空闲，视频解码停止 |
| 24 小时内存 | 工作集无单调增长，末值不超过稳态基线 15% |
| 退出 | 无 Project D 残留进程；独立恢复看门程序按设计退出 |

4K、多屏和不同显卡使用同一测量方法建立独立基线，不把轻薄本阈值机械套用到所有硬件。

## Gate 3：兼容性实验室

目标：用矩阵证据替代“理论支持”。

最低矩阵：

- Windows 10 22H2、Windows 11 当前受支持版本。
- Intel 核显、AMD 核显/独显、NVIDIA 独显。
- 1080p、2K、4K、竖屏、超宽屏。
- 100%、125%、150%、200% 缩放。
- 单屏、双屏、三屏；同分辨率和混合分辨率；60/120/144Hz 混合刷新率。
- 笔记本电池、外接显示器热插拔、Explorer 重启、休眠/唤醒、锁屏/解锁。

关闭条件：

- 每个矩阵项保存机器信息、安装包哈希、操作步骤、截图/录像和日志。
- 20 次睡眠/唤醒、50 次显示器插拔或切换，无 P0/P1。
- DPI 切换后图标、容器、菜单、桌宠和设置页无裁剪、错位或不可点击区域。
- Windows/显卡驱动更新前后通过同一核心回归。

缺少真实设备时可使用云 Windows/虚拟机覆盖安装和普通 UI，但 GPU、视频解码、多屏、DPI 和睡眠必须使用物理设备。

## Gate 4：可信安装、更新和回滚

目标：让每次版本发布都可验证、可恢复、可追踪。

交付项：

- 使用有效的 Windows 代码签名证书签署安装包、卸载器和更新包。
- 建立稳定/测试两个更新通道，更新清单包含版本、SHA-256、签名和最低数据库版本。
- 更新下载到临时目录，完成哈希与签名校验后再替换；断网或损坏不影响当前版本。
- 升级前复制数据库并校验；启动失败自动回滚应用版本和数据库备份。
- 验证全新安装、覆盖安装、跨版本迁移、中文路径、标准账号、无网安装和卸载。
- 卸载流程明确“保留个人数据/彻底删除”，两种路径都恢复桌面状态。
- CI 生成 SBOM、依赖审计、secret scan、安装包哈希和发布说明。

关闭条件：

- 至少完成 N-2 到 N、N-1 到 N、损坏包、断网、磁盘不足和迁移失败回滚。
- 所有发布二进制 Authenticode 有效且发布哈希可公开核对。
- Defender 与至少一种第三方杀软无阻止安装的高风险误报。

## Gate 5：账号、订单、支付与权益

目标：建立不能由客户端自说自话的商业闭环。

服务端最小模块：

- Account：注册、登录、退出、找回、会话吊销和设备列表。
- Device：设备注册、数量限制、换机解绑和异常登录审计。
- Order：订单号、金额、币种、渠道、状态机、幂等键和客服查询。
- Payment：微信/支付宝等渠道下单、服务端验签、异步回调、补单、超时和退款。
- Entitlement：套餐、开始/到期、永久权益、撤销、签名快照和离线宽限期。
- Audit：订单、权益和管理员动作的不可覆盖审计记录。

安全规则：

- 客户端只发起订单和查询结果，不判断支付成功。
- 回调按渠道订单号与幂等键处理，重复回调不得重复开通。
- 支付密钥、AI 公共密钥和管理员接口只存在于服务端密钥系统。
- 本地 entitlement 仅作为短期签名缓存；篡改数据库不能解锁会员。
- 退款、拒付和风险订单会撤销权益并保留审计记录。

关闭条件：

- 支付成功、失败、取消、超时、重复回调、延迟到账、退款和断网补单全通过。
- 换机恢复、设备超限、会员到期、永久会员和离线宽限全通过。
- 完成服务端渗透测试、限流、防重放、密钥轮换和灾备演练。

## Gate 6：内容版权、隐私与运营

目标：让产品内容和数据处理经得起商业审查。

交付项：

- 建立资产台账：`asset_id`、作者、原始地址、下载日期、授权类型、许可快照、证明哈希、版本和上下架状态。
- 用户提供和 AI 生成资源分别保存权利保证、模型条款、提示词、生成日期和人工审核结果。
- 建立侵权投诉、证据保全、暂停分发、下架、替换和申诉流程。
- 发布隐私政策、用户协议、会员协议、自动续费说明和儿童/敏感信息边界。
- 诊断上传保持显式同意、字段白名单、脱敏和保留期限。
- 建立崩溃率、更新成功率、支付到账率和恢复成功率仪表板；不上传桌面文件名和完整路径。

关闭条件：

- 商业包内每个资源均能从安装包文件追溯到授权证明。
- 隐私政策与实际采集字段一致，账号删除和数据导出服务端闭环通过。
- P0 有值班、止损、回滚和用户通知模板。

## Gate 8：发布后运行保障

目标：确保上线后发生问题时可快速发现、止损和恢复。

交付项：

- 版本开关和远程禁用高风险功能。
- 更新通道暂停能力。
- 问题版本撤回（强制停止分发）。
- 强制停止分发某个壁纸资产。
- 支付渠道降级。
- 服务状态页。
- 客服工单流程。
- P0 事件响应规范（负责人、响应时间、复盘模板）。
- 用户通知模板。
- 崩溃版本看板。
- 更新失败率告警。
- 支付到账率告警。
- 权益服务异常告警。
- 数据库容量和备份告警。

门禁指标：

- 能在不发布新客户端的情况下停用问题资源或关闭高风险在线功能。
- 能在发现问题后停止问题版本继续扩散。
- 能定位受影响版本、用户和订单。
- P0 事件有负责人、响应时间和复盘模板。
- 壁纸核心引擎不建议远程随意修改，但在线内容、支付入口、更新分发、天气和部分 AI 能力应具备止损开关。

## Gate 7：Release Candidate 与公开发布

目标：只发布经过证据审查的构建。

RC 阶段：

1. RC0：团队内测，关闭全部已知 P0。
2. RC1：50 至 100 人封闭 Beta，验证硬件分布、崩溃与更新。
3. RC2：500 人灰度，验证支付、带宽、客服和回滚。
4. GA：P0 为 0，P1 有明确豁免责任人与截止日期，签名和合规材料齐全。

GA 核心指标：

- 无法启动、壁纸黑屏、桌面无法恢复、支付错误和数据丢失均为 0 个已知未解决 P0。
- 7 日无崩溃会话率达到 99.5% 以上。
- 更新成功率达到 99% 以上，失败用户可自动留在旧版本。
- 支付成功后的权益到账率达到 99.9%，其余订单可自动补偿或人工追踪。
- 24 小时长测和核心兼容矩阵全部有对应构建哈希的证据。

## 5. 架构推进

V3 保留现有 Electron + Vue + TypeScript + sql.js 基线，不因为商业化重写客户端。

建议继续拆分：

- `WindowLifecycleService`：窗口创建、健康探针、崩溃恢复和安全重启。
- `WallpaperRuntime`：每屏媒体实例、暂停仲裁、资源释放和性能档位。
- `PowerAndPresenceService`：锁屏、休眠、电池、全屏和恢复事件。
- `UpdateService`：通道、下载、签名验证、安装和回滚状态。
- `MetricsService`：本地性能与稳定性报告，默认不上传隐私数据。
- `AccountClient` / `EntitlementClient`：服务端会话和签名权益，和本地业务能力解耦。

数据库新增应通过版本迁移：

- `media_assets`
- `display_wallpaper_assignments`
- `runtime_metrics`
- `update_history`
- `account_sessions`
- `entitlement_cache`

任何迁移继续执行“备份在先、临时库校验、原子替换、失败回滚”。

## 6. 推荐执行顺序与投入

| 波次 | 内容 | 预计工程投入 | 商业准备度目标 |
|---|---|---:|---:|
| W0 | Gate 0 外壳可靠性收尾 | 3-5 工程日 + 真实睡眠测试 | 40% |
| W1A | Gate 1A 媒体运行时（PauseArbiter + 视频生命周期 + 解码降级）| 1.5-2 周 | 48% |
| W1B | Gate 1B 多屏与系统恢复（每屏映射 + 自启动 + Explorer 恢复）| 1-1.5 周 | 52% |
| W2 | Gate 2 性能采样工具与长测脚本 | 1 周 | 56% |
| W2+W3 | Gate 2/3 并行推进：每增加一种硬件同时产出 Gate 2 基线和 Gate 3 兼容性证据 | 2-4 周（硬件可用时并行）| 70% |
| W4 | Gate 4 签名更新、回滚和安装矩阵 | 2-3 周 | 80% |
| W5 | Gate 5 账号支付权益服务端 | 4-6 周 | 90% |
| W6 | Gate 6 合规与运营体系 | 2-4 周，可并行 | 96% |
| W7 | Gate 7 RC0-RC2 | 2-3 周 | 100% 发布门禁 |
| W8 | Gate 8 发布后运行保障 | 持续，与 W6-W7 并行 | GA 后运维 |

单人开发的合理关键路径约 14-20 周；支付渠道审核、代码签名证书、法律文本、授权取证和物理硬件测试不完全受开发速度控制。不能用压缩测试时间换取表面进度。

**关键约束：** Gate 2（性能采样工具）先于 Gate 3（兼容性矩阵）交付。每增加一种新硬件时，同时产出 Gate 2 性能基线和 Gate 3 兼容性证据。不在单机上宣布性能门禁关闭后再做兼容性。

## 7. 需要人工或外部资源

以下工作无法仅靠本机代码完成：

- 真实双屏/三屏、AMD/NVIDIA、4K、竖屏和 Windows 10 设备。
- 20 次睡眠/唤醒与 24 小时时间窗口。
- Windows 代码签名证书及主体认证。
